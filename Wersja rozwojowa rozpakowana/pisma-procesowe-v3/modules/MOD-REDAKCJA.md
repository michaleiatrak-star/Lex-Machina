# MOD-REDAKCJA — Redakcja i Poprawa Istniejącego Pisma

*Ładuj gdy: użytkownik dostarcza GOTOWE pismo (własne lub przeciwnika do analizy
stylu, nie do obalania merytorycznego — to jest MOD-OBAL) i prosi o: "popraw",
"skróć", "wzmocnij", "przeredaguj", "poprawi styl", "zrób bardziej stanowcze/
neutralne/negocjacyjne", "sprawdź i wygładź", "czy to dobrze napisane".*

---

## R.0 — ODRÓŻNIENIE OD MODELU W1-W2-W3

```
MOD-REDAKCJA NIE jest substytutem modelu trzech wiadomości.
Stosuj go WYŁĄCZNIE gdy WSZYSTKIE poniższe są prawdą:

✓ Użytkownik dostarczył TREŚĆ ISTNIEJĄCEGO pisma (wklejona, plik, .docx)
✓ Pismo ma już swoją argumentację, strukturę i przepisy/cytaty
✓ Żądanie dotyczy FORMY/STYLU/DŁUGOŚCI/TONU, nie nowej strategii procesowej
  od zera

Jeśli którykolwiek warunek NIE jest spełniony → wróć do KROK 0 (Test B/C)
modelu W1-W2-W3. W szczególności:
  ✗ "napisz mi odpowiedź na ten pozew" → W1-W2-W3 (nowe pismo)
  ✗ "to pismo przeciwnika — jak je obalić" → MOD-OBAL + W1-W2-W3
  ✗ "dodaj nowy zarzut oparty na art. X" → wymaga W1 (nowa podstawa prawna
    = nowa weryfikacja ISAP w pełnym modelu, nie tylko redakcja stylu)

MOD-REDAKCJA może być WYWOŁANY WEWNĄTRZ W2 jako krok pomocniczy (redakcja
draftu przed przejściem do W3), ale jako samodzielna ścieżka służy do
PRACY NA GOTOWYM TEKŚCIE bez przechodzenia automatu stanów od początku.
```

## R.1 — INTAKE (jedno pytanie zbiorcze, jeśli brakuje danych)

```
□ TREŚĆ PISMA: [wklejona / plik — jeśli plik, wczytaj przed dalszymi krokami]
```

Jeśli użytkownik nie wskazał TONU i CO POPRAWIĆ jest ogólne ("popraw to

(R.3), a na końcu krótko zaznacz jakie ustawienia przyjęto i że można zmienić.

## R.2 — PARAMETR TON

Cztery nazwane tony, wybierane przez użytkownika lub domyślne. Ton wpływa na
DOBÓR SFORMUŁOWAŃ, nie na fakty, przepisy, żądania ani strukturę pisma —
te pozostają niezmienione (lub zmieniane wyłącznie w ramach R.3/R.4, niezależnie
od tonu).

```
TON: STANOWCZY
  Charakterystyka: zdania krótkie, twierdzące, czas teraźniejszy i przeszły
  dokonany ("Pozwany nie wykonał...", "Powód wykazał..."), minimalna
  modalność ("wnoszę o" zamiast "uprzejmie proszę o rozważenie").
  Unikaj: zwrotów łagodzących ("być może", "wydaje się, że", "z całym
  respektem"), wielokrotnych zastrzeżeń.
  Kiedy: pisma procesowe w sporze, riposty, wezwania ostateczne, sytuacje
  gdy strona przeciwna działa w złej wierze lub eskaluje.
  Ryzyko nadużycia: zbyt stanowczy ton przy braku pokrycia dowodowego
  brzmi nieprofesjonalnie — MOD-REDAKCJA NIE wzmacnia tonu twierdzeń, które
  w R.4/QUALITY-CHECK są oznaczone jako gołosłowne; w takim przypadku
  zachowaj ostrożniejsze sformułowanie i odnotuj to w raporcie zmian.

TON: NEUTRALNY (DOMYŚLNY)
  Charakterystyka: standardowy ton procesowy — rzeczowy, bez emocji,
  zgodny z MOD-WALIDACJA BLOK C ("brak ocen moralnych", "brak sformułowań
  emocjonalnych"). To jest ton, do którego W3.4/BLOK C i tak doprowadza
  każde pismo — MOD-REDAKCJA w tym trybie głównie usuwa odchylenia od niego.
  Kiedy: większość pism procesowych, korespondencja z sądem/organem,
  sytuacje gdy nie ma wskazania do innego tonu.

TON: NEGOCJACYJNY
  Charakterystyka: zdania łączące stanowisko z otwartością — "Strona X
  wnosi o..., jednocześnie pozostając otwarta na ustalenie...", warunkowość
  bez utraty mocy żądania, unikanie sformułowań które zamykają drogę do
  ugody (np. "wyłącznie", "bezwzględnie", "w żadnym wypadku").
  Kiedy: korespondencja przedsądowa, propozycje ugodowe, pisma do
  kontrahenta przed eskalacją, sytuacje gdy ugoda jest realną opcją.
  Uwaga: ton negocjacyjny w PIŚMIE PROCESOWYM (do sądu) może być
  niewłaściwy — jeśli MOD-REDAKCJA wykryje że dokument to pismo procesowe
  (nagłówek z oznaczeniem sądu/sygnatury) a użytkownik prosi o ton
  negocjacyjny, zapytaj czy chodzi o ZAŁĄCZONĄ PROPOZYCJĘ UGODOWĄ (osobny
  dokument) czy o samo pismo procesowe — nie zmiękczaj petitum/żądań
  pisma do sądu bez wyraźnego potwierdzenia.

TON: ZWIĘZŁY (SKRÓCONY)
  Charakterystyka: nie jest to odrębny "głos", lecz instrukcja kompresji:
  usuń powtórzenia, połącz zdania o tej samej tezie, usuń przykłady
  poboczne, zachowaj WSZYSTKIE elementy z R.3 listy obowiązkowej
  (żądania, podstawy prawne, fakty kluczowe). Może być kombinowany z
  każdym z trzech tonów wyżej (np. "stanowczy i zwięzły").
  Kiedy: użytkownik explicit prosi o "krócej", "skróć", "za długie".
```

**Format odnotowania w odpowiedzi:**
```
Ton zastosowany: [STANOWCZY/NEUTRALNY/NEGOCJACYJNY] [+ ZWIĘZŁY jeśli dotyczy]
[jeśli DOMYŚLNY i nie wskazany explicit:] (nie wskazano tonu — przyjęto
NEUTRALNY; daj znać jeśli wolisz inny)
```

## R.3 — ZAKRES ZMIAN: CO JEST "BEZPIECZNE" A CO WYMAGA ZGODY

```
MOD-REDAKCJA MOŻE bez pytania:
  ✓ poprawiać styl, gramatykę, interpunkcję, powtórzenia
  ✓ skracać poprzez kompresję (R.2 ZWIĘZŁY) — bez usuwania treści merytorycznej
  ✓ zmieniać kolejność akapitów dla lepszej logiki (bez zmiany kolejności
    żądań w petitum, jeśli to pismo procesowe — petitum ma własną kolejność
    procesową, np. żądanie główne przed ewentualnym)
  ✓ wzmacniać/łagodzić ton zgodnie z R.2
  ✓ poprawiać formułowanie wniosków zgodnie z QUALITY-CHECK
    ("wnoszę o sprawiedliwe rozstrzygnięcie" → wniosek precyzyjny,
    JEŚLI z kontekstu pisma wynika co konkretnie strona chce — nie
    wymyślaj nowego żądania)

MOD-REDAKCJA NIE MOŻE bez wyraźnej zgody użytkownika:
  ✗ usuwać żądań, zarzutów, dowodów lub argumentów merytorycznych
  ✗ dodawać nowe przepisy, orzeczenia lub fakty (to wymaga W1 + weryfikacji
    ISAP — poza zakresem tego modułu)
  ✗ zmieniać kwoty, daty, oznaczenia stron, sygnatury
  ✗ zmieniać kwalifikację prawną zarzutu

Jeśli w trakcie redakcji MOD-REDAKCJA zauważy, że usunięcie czegoś
POPRAWIŁOBY pismo (np. argument szkodliwy, sprzeczność wewnętrzna,
przyznanie niekorzystne) — NIE usuwaj automatycznie. Odnotuj w RAPORCIE
ZMIAN (R.5) jako "SUGESTIA — wymaga decyzji", z krótkim wyjaśnieniem.
Jeśli sprzeczność jest poważna (np. wewnętrzna sprzeczność dat/faktów),
zasygnalizuj możliwość uruchomienia analizator-dowodow-v3/MD3c.
```

## R.1b — TEZA-GATE (obowiązkowy, przed KROK 2 — naprawa F-7/ZASADA 11)

```
⛔ Zanim MOD-REDAKCJA zdiagnozuje styl/ton (KROK 2), zrekonstruuj JEDNYM
ZDANIEM tezę centralną dostarczonego pisma: czego strona żąda i na jakiej
podstawie prawnej/faktycznej. Zapisz to jawnie w odpowiedzi (1 linijka),
NIE tylko w myślach.

Powód: bez tej rekonstrukcji MOD-REDAKCJA może np. wzmocnić ton twierdzenia,
które przy uważnym czytaniu tezy okazuje się słabo poparte dowodowo, albo
"wygładzić" fragment, który w rzeczywistości ukrywa lukę w argumentacji —
redagując czysto stylistycznie bez zauważenia problemu merytorycznego,
który wymagałby SUGESTII w R.5, nie cichej poprawki kosmetycznej.

Jeśli teza centralna NIE jest możliwa do ustalenia jednym zdaniem (pismo
wielowątkowe, kilka roszczeń) → wypisz tezy osobno dla każdego wątku, nie
uśredniaj ich do jednego ogólnika.

To nie jest ponowna weryfikacja merytoryczna (to rola W1-W2-W3/MOD-OBAL) —
to jednozdaniowe stwierdzenie faktu, służące jako punkt odniesienia dla
KROK 2 i R.3 (żeby "bezpieczne" zmiany stylu nie kolidowały cicho z tym,
co pismo faktycznie twierdzi).
```

## R.4 — PRZEBIEG REDAKCJI

```
KROK 1 — Wczytaj treść pisma (jeśli plik: użyj odpowiedniego skilla
         file-reading/docx, NIE kopiuj treści ręcznie z OCR jeśli plik
         jest dostępny natywnie)

KROK 2 — Szybka diagnoza (bez generowania jeszcze nowej treści):
         - Czy to pismo procesowe (sąd/organ) czy korespondencja
           (kontrahent/strona)? → wpływa na R.2 ograniczenia tonu
         - Jaki jest aktualny ton? (już stanowczy / już neutralny / mieszany)
         - Czy są oczywiste naruszenia QUALITY-CHECK / BLOK C? (oceny moralne,
           sformułowania emocjonalne, ogólniki, wnioski nieprecyzyjne)
         - Czy są oznaki sprzeczności wewnętrznych (daty, kwoty, nazwy)?
           → jeśli TAK i istotne, odnotuj do R.5, nie naprawiaj w locie
             bez wskazania

KROK 3 — Wczytaj moduły wsparcia:
         view /mnt/skills/user/shared/QUALITY-CHECK.md
         view /mnt/skills/user/shared/MOD-WALIDACJA_v2.md   (sekcja BLOK C)
         Jeśli pismo procesowe i użytkownik prosi o sprawdzenie formalne
         (zob. R.6) → dodatkowo:
         view /mnt/skills/user/shared/FORMAL-CHECK.md

KROK 4 — Wygeneruj poprawioną wersję, zachowując:
         - wszystkie żądania/wnioski (petitum) w niezmienionej kolejności
           procesowej, jeśli pismo procesowe
         - wszystkie przepisy/orzeczenia/sygnatury BEZ ZMIAN (MOD-REDAKCJA
           nie weryfikuje ich ponownie — to nie jest W3; jeśli użytkownik
           chce dodać/zmienić przepis, przekieruj do W1 pełnego modelu)
         - wszystkie daty, kwoty, oznaczenia stron BEZ ZMIAN

KROK 5 — Sporządź RAPORT ZMIAN (R.5)
```

## R.4b — GATE JAKOŚCI (obowiązkowy po R.4, przed R.5)

> Ścieżka MOD-REDAKCJA (Test A) omija W1–W3, ale nie zwalnia z minimalnej
> weryfikacji jakościowej. Wykonaj poniższe kroki po zakończeniu redakcji
> tekstu, a przed sporządzeniem R.5.

```
KROK G1 — PODMIOT-GATE (uproszczony):
  Zidentyfikuj podmioty w redagowanym piśmie (nazwy spółek, sądy, organy).
  Czy jakikolwiek podmiot prowadzący działalność ma oznaczenie ⚠️POD
  lub czy jego dane budzą wątpliwość?
    TAK → wykonaj W3.0 PODMIOT-GATE dla tych podmiotów
          (KRS/CEIDG/BIP — jak w pełnym W3.0)
    NIE → odnotuj "PODMIOT-GATE: N/D — brak podmiotów wymagających weryfikacji"

KROK G2 — AUDYT-KONCOWY (uproszczony — 2 kategorie):
  Oceń zredagowane pismo w dwóch najważniejszych kategoriach:
    Spójność petitum z uzasadnieniem:  __/10
    Realizm sądowy (ton, proporcje):   __/10
  Jeśli którakolwiek < 7 → wskaż poprawkę, wykonaj ją, powtórz ocenę.
  Nie blokuj całości .docx za te dwie kategorie — tylko sygnalizuj w R.5.

KROK G3 — HYBRID-VALIDATION (auto-raport braków formalnych):
  view /mnt/skills/user/shared/HYBRID-VALIDATION.md
  FAZA 1: wylistuj braki 🔴/🟡/🔵 bez pytania o zgodę.
  FAZA 2: jeśli użytkownik dostarczył dane → wstaw. Jeśli nie → ⬛.
  Braki 🔴 blokują oznaczenie pisma jako gotowego.
```

## R.6 — RELACJA DO "3 TRYBÓW SPRAWDZANIA" (przewodnik-prawny-v2 KROK F)

```
MOD-REDAKCJA wykonuje ZMIANY w tekście. KROK F w przewodnik-prawny-v2 wykonuje
OCENĘ/RAPORT bez zmiany tekstu (3 tryby: formalny / merytoryczny / procesowy).

Jeśli użytkownik prosi o "sprawdź to pismo" (bez słowa "popraw"/"zredaguj") →
to jest KROK F (raport), NIE MOD-REDAKCJA.

Jeśli użytkownik prosi o "sprawdź i popraw" / "sprawdź formalnie i wygładź
styl" → wykonaj odpowiedni tryb z KROK F NAJPIERW (raport), a następnie —
jeśli użytkownik potwierdzi lub od razu prosił o poprawki — MOD-REDAKCJA
na podstawie ustaleń z KROK F (np. tryb formalny wykrył brak podpisu →
MOD-REDAKCJA nie "dopisuje" podpisu, ale może dodać placeholder [PODPIS]
i odnotować w R.5).

Połączenie typowe:
  "sprawdź formalnie i popraw styl" →
    1. KROK F.4 (walidacja formalna) → raport braków formalnych
    2. MOD-REDAKCJA (zakres: styl, R.3) → poprawiona treść + R.5
    3. Braki formalne z kroku 1, które NIE są stylem (np. brak załącznika)
       → wymień osobno jako "do uzupełnienia przez użytkownika", nie część
       redakcji stylu
```

## INTEGRACJA

| Sytuacja | Idź do |
|---|---|
| Pismo własne, gotowe, chcę poprawić styl/ton/długość | MOD-REDAKCJA (ten moduł) |
| Pismo przeciwnika, chcę je obalić merytorycznie | MOD-OBAL + model W1-W2-W3 |
| Nowe pismo od zera | KROK 0 → Test B/C → W1-W2-W3 lub pisma-proste-v2 |
| Chcę tylko raport (bez zmian) — formalny/merytoryczny/procesowy | `przewodnik-prawny-v2` KROK F |
| Wykryto sprzeczność wewnętrzną podczas redakcji | zasygnalizuj `analizator-dowodow-v3` MD3c (nie naprawiaj sam) |
| Redakcja w trakcie W2 (przed W3) | MOD-REDAKCJA jako pomocniczy krok wewnątrz W2, R.3 obowiązuje identycznie |
